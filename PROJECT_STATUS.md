# Whisky Scanner 現状整理

## 1. 現在の実装状況

### 使用技術スタック

- Frontend
  - Next.js 14
  - React 18
  - TypeScript
  - App Router
  - `globals.css` による素のCSS

- Backend
  - FastAPI
  - Python 3.12
  - SQLAlchemy
  - psycopg
  - pydantic-settings

- Database
  - PostgreSQL 16 alpine

- Infrastructure
  - Docker / Docker Compose
  - `frontend`, `backend`, `db` の3コンテナ構成

### 実装済み画面

現在、画面は `frontend/app/page.tsx` のトップページのみ。

内容は以下。

- Whisky Scanner の簡易ヒーロー表示
- API / DB の接続ステータス表示
- MVPスコープの簡単な説明表示

ただし、日本語テキストが文字化けしているため、UX以前に表示文言として読めない状態。

### 実装済み機能

- フロントエンドからバックエンドの `/health` を呼び出し
- バックエンドでDB接続確認
- PostgreSQLへの `SELECT 1` による疎通チェック
- Docker Composeでフロント、API、DBをまとめて起動できる構成
- OCR用サービスのインターフェース雛形
  - `OCRService`
  - `DummyOCRService`

### API

実装済みエンドポイントは1つ。

- `GET /health`

レスポンス例:

```json
{
  "status": "ok",
  "database": "ok"
}
```

DB接続に失敗した場合は概ね以下。

```json
{
  "status": "degraded",
  "database": "error"
}
```

## 2. プロジェクトの目的の推測

このプロジェクトは、ウイスキー好きユーザーがボトル画像を起点にウイスキー情報へアクセスし、飲んだ記録や次の一本との出会いにつなげるアプリを目指していると考えられる。

現状の名前は `Whisky Scanner` で、README上も「ラベル画像をアップロードし、OCRで候補ボトルを返すMVP」を目指している様子。

想定される中核体験は以下。

1. ユーザーがボトルやラベルの写真をアップロードする
2. OCRで銘柄名・蒸留所名・年数・表記を読み取る
3. 該当しそうなウイスキー候補を表示する
4. ユーザーが飲んだ記録を残す
5. 味の傾向や過去の好みから、似た一本・次に試す一本を提案する
6. 必要に応じて購入導線につなげる

テーマとしては以下の3つが中心。

- 飲んだ記録
- 次に似た一本との出会い
- 購入支援

## 3. 現状コードから見たUX上の課題

### 画面がまだ「サービス体験」ではなく「疎通確認」になっている

現在のトップページはAPI/DBの接続確認が主目的で、ユーザーがウイスキーアプリとして何をすればよいかはまだ伝わらない。

本来のファーストビューでは、以下のような行動が主役になるべき。

- ボトル写真を撮る / アップロードする
- 飲んだウイスキーを記録する
- 似たウイスキーを探す

### 日本語文言が文字化けしている

READMEと画面内の日本語が文字化けしている。これは企画共有、開発引き継ぎ、ユーザー体験のすべてに悪影響があるため、早めに直すべき。

### ドメインモデルが未定義

「ウイスキーアプリ」として重要な概念がまだコードに存在しない。

不足している主な概念:

- Whisky / Bottle
- Distillery
- Region
- Age
- ABV
- Cask type
- Flavor notes
- Tasting record
- Rating
- Similarity / Recommendation
- Purchase link / Shop

UXを固めるには、先に「何を記録し、何を比較し、何を推薦するか」をデータ構造として決める必要がある。

### OCR起点の体験が未実装

`DummyOCRService` はあるが、実際のOCRもアップロード導線もない。
現状では「スキャナー」というプロダクト名に対して、主要体験がまだ存在しない。

### 記録アプリとしての継続利用導線がない

飲んだ記録、履歴、好みの蓄積、再訪したくなる画面が未実装。
このままだと単発の検索ツールに寄りやすい。

### 推薦の根拠がまだ設計されていない

「似た一本」と言うためには、何を似ていると判断するかが必要。

例:

- 味わい
- 香り
- 樽
- 地域
- 蒸留所
- 価格帯
- アルコール度数
- 飲み方
- ユーザー評価

現状コードには、その判断材料がまだない。

## 4. 次に着手すべき優先タスク

### 小タスク

すぐ着手でき、影響範囲が小さいもの。

- READMEと画面文言の文字化け修正
- トップページの文言を「ウイスキー体験」向けに書き換える
- API接続状態表示を開発用表示に寄せる、または目立たない位置へ移動する
- `.env.example` の説明をREADMEに整理
- `GET /health` のレスポンス型を明示
- フロント側に簡易ローディング / エラー表示を追加
- OCRダミーの返却値を空文字ではなくサンプルテキストにする

- 画像アップロードAPIを追加

- ボトル詳細の基本データ構造を作成

  - `distilleries`

- DBマイグレーション導入
- 飲んだ記録の作成・一覧表示
- 味わいタグや評価の入力UIを作る

### 大タスク

プロダクト価値に直結するが、設計が必要なもの。

- OCR精度向上
  - ラベル画像の前処理

  - 曖昧検索
- 類似ウイスキー推薦ロジック
  - 味わいベース
  - 樽・地域・価格帯ベース
  - ユーザーの記録ベース
- 購入支援
  - 外部ECリンク
  - 価格比較
  - 在庫情報
  - お気に入り / 欲しいリスト

  - 自分の記録
  - 好み分析

## 5. 推奨する次の進め方

まずは「スキャンして候補が出る」体験をMVPの中心に置くのがよさそう。

優先順は以下。

1. 文字化け修正
2. トップ画面をユーザー向けに再設計
3. 画像アップロードUI追加
4. 画像アップロードAPI追加
5. OCRダミー結果を使って候補表示
6. ウイスキーの最小データモデル作成
7. 飲んだ記録機能追加
8. 類似推薦ロジックの仮実装

現状はまだ「起動する土台」ができた段階。
次の一手は、技術的な完成度よりも「ユーザーがこのアプリで何をできるのか」が一目でわかる体験を作ること。
